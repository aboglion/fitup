מסמך אפיון מערכת (PRD & System Architecture) מלא, מפורט ומעודכן עבור **אפליקציית אימונים חכמה (Offline-First PWA)**.

---

## 1. תמצית ומטרות המערכת

אפליקציית אימונים הפועלת במודל Progressive Web App (PWA) צד-לקוח (Client-Side). המערכת מחליפה ניהול ידני של תוכניות אימון באלגוריתם אוטומטי המבוסס על עקרונות ה-Double Progression. המערכת פועלת באופן עצמאי ללא תלות בשרת, מגבה נתונים בחשבון גוגל דרייב של המשתמש, ותומכת באופן מלא בסופר-סטים, יומן מעקב חכם, והגדרות לוגיקה גמישות.

---

## 2. ארכיטקטורה טכנולוגית (Tech Stack)

| רכיב מערכת | טכנולוגיה | תפקיד ופונקציונליות |
| --- | --- | --- |
| **ממשק משתמש (UI)** | HTML5, CSS3, Vanilla JavaScript | רינדור המסכים, הפעלת טיימרים חיים (קצב/מנוחה), וניהול אינטראקציית המשתמש. |
| **אריזת מובייל (PWA)** | `manifest.json`, Service Worker | מאפשר התקנה על מסך הבית וטעינת נכסים וקוד גם ללא חיבור לאינטרנט (Offline Mode). |
| **מסד נתונים מקומי** | `localStorage` | מקור האמת המיידי (Source of Truth). מבטיח זמן תגובה של 0 מילי-שניות (Zero Latency) באימון. |
| **גיבוי וסנכרון ענן** | Google Drive REST API (OAuth 2.0) | סנכרון ושמירת ה-JSON המלא לתיקיית `appDataFolder` מוסתרת בדרייב כגיבוי מאובטח. |
| **אירוח (Hosting)** | GitHub Pages / Vercel / Cloudflare Pages | אירוח חינמי לחלוטין של קבצי המקור הסטטיים. |
| **מנוע AI (אופציונלי)** | Chrome `window.ai` / External API | ניתוח נתונים אוטומטי. מתבסס על המודל המובנה בדפדפן (ללא עלות) או הזנת מפתח API אישי בהגדרות. |

## 4. מנוע לוגיקה ואלגוריתמיקה (Progression & Rules Engine)

הלוגיקה מופעלת פנימית באמצעות JavaScript עם סיום כל בלוק אימון.

### 4.1 אלגוריתם ה-Double Progression

עבור כל תרגיל שהוגדר תחת מודל זה, המערכת בודקת את התנאים הבאים בסיום האימון:

* **תנאי שדרוג (Level Up):**
1. הושלמו כל הסטים המוגדרים ב-`target_sets`.
2. מספר החזרות בכל סט הגיע לרף העליון של `target_reps_range` (לדוגמה: 10 מתוך 6-10).
3. בכל הסטים התקיים תנאי `strict === true`.
4. בכל הסטים התקיים התנאי הגלובלי `rir >= minimum_rir_for_progression`.


* **פעולת השדרוג (Reset & Build):** שדרוג ה-`current_level_color` (למשל מצהוב לירוק), איפוס ספירת הכשלים (`consecutive_fails = 0`), והורדת יעד החזרות לאימון הבא לרף התחתון של הטווח (לדוגמה: ירידה חזרה ל-6 חזרות).
* **פעולת נסיגה (Deload):** אם תנאי השדרוג לא מתקיימים במשך מספר אימונים העולה על `failed_workouts_trigger_deload`, המערכת מורידה את יעד הסטים/חזרות או מתריעה בפני המשתמש לשנות וריאציה.

### 4.2 לוגיקת סופר-סטים (Supersets)

כאשר המערכת קוראת מ-`routines_templates` בלוק מסוג `superset`:

1. טיימר המנוחה מושבת לאחר התרגיל הראשון.
2. הממשק מחליף מיד לתרגיל השני.
3. רק לאחר סיום התרגיל האחרון בשרשרת הסופר-סט, מופעל הטיימר המוגדר תחת `timers.compound_rest_sec`.
4. אלגוריתם ה-Progression רץ ומחשב התקדמות באופן עצמאי ומבודד לכל תרגיל בתוך הסופר-סט.

## 5. ממשק משתמש וזרימת מסכים (UI / UX Flow)

### מסך 1: Dashboard & Calendar (מסך הבית)

* **חיווי ענן:** תצוגה עליונה של סטטוס הסנכרון (מעודכן / ממתין לרשת / אופליין).
* **יומן שבועי:** תצוגת לוח שנה המזהה את היום הנוכחי ומושכת את התוכנית הרלוונטית מ-`calendar_and_cycles`.
* **כפתור פעולה ראשי:** "התחל אימון [שם התוכנית]".

### מסך 2: Active Workout (האימון הפעיל)

* **תצוגת תרגיל:** כותרת, צבע הרמה הנוכחית, ויעד הסטים/חזרות להיום.
* **טיימרים ויזואליים:** * **Tempo Timer:** אנימציה נושמת המכתיבה את זמני השלב האקסצנטרי, העצירה והכיווץ.
* **Rest Timer:** ספירה לאחור אוטומטית המופעלת בסיום סט או בסיום בלוק סופר-סט, מלווה בחיווי קולי בסיומה.



### מסך 3: Data Logging Modal (חלון הזנת נתונים)

מופיע אוטומטית בזמן טיימר המנוחה:

* בורר חזרות (`+` / `-`) המכויל מראש ליעד הנדרש.
* סליידר צבעוני להזנת RIR (מ-0 אדום עד 5 ירוק).
* תיבת סימון (Checkbox) לאישור ביצוע בטכניקה מחמירה.

### מסך 4: Settings & Configuration (הגדרות)

* **התחברות גוגל (OAuth):** כפתור אימות לחיבור חשבון לטובת גיבוי לדרייב.
* **ניהול חוקים:** ממשק (Forms) לעדכון ערכי ה-`system_settings` (זמני מנוחה דיפולטיביים, חוקי התקדמות).
* **מנהל יומן ותבניות:** אפשרות לערוך את מבנה הימים והבלוקים ללא צורך בכתיבת קוד.
* **ייצוא/ייבוא:** כפתורים ידניים להורדת קובץ ה-JSON למכשיר או העלאת קובץ קיים.

## 6. פרוטוקול אופליין וסנכרון רשת (Offline-First Sync Logic)

המערכת פועלת תחת חוק ברזל: **הממשק לא ממתין לענן לעולם.**

1. **אתחול המערכת (App Load):**
* טעינת האפליקציה בודקת קודם כל את ה-`localStorage`.
* אם הזיכרון ריק (מכשיר חדש), המערכת דורשת אינטרנט כדי למשוך את קובץ ה-`workout_data.json` מגוגל דרייב (`appDataFolder`).
* מרגע שהקובץ נטען מקומית, ניתן לעבור למצב אופליין מלא.


2. **זרימת נתונים בעת אימון:**
* המשתמש מזין תוצאה בסט.
* האפליקציה מעדכנת את אובייקט ה-JSON ושומרת אותו מיידית ב-`localStorage`. הדגל `sync.is_dirty` הופך ל-`true`.


3. **תהליך הסנכרון ברקע (Background Sync):**
* המערכת מאזינה לאירועי רשת של הדפדפן (`window.addEventListener('online', ...)`).
* ברגע שיש חיבור פעיל והדגל `is_dirty` פועל, פונקציה אסינכרונית משגרת בקשת `PATCH` ל-Google Drive API.
* לאחר אישור ההעלאה, הדגל `is_dirty` חוזר ל-`false` והתאריך ב-`last_cloud_sync` מתעדכן. המערכת חוזרת למצב גיבוי מסונכרן.
